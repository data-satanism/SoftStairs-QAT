## 1. Что означает оставить low-rank и sparse как отдельные ветви
Про реализацию метода см. п 3.3 раздел про инференс в документе [«Анализ гипотез по SoftStairs, LR-QAT и структурированным градиентам»](<./Анализ гипотез по SoftStairs, LR-QAT и структурированным градиентам.md>).

**Кастомный CUDA kernel не обязателен**. Его имеет смысл писать только тогда, когда обычная реализация через PyTorch оказывается слишком медленной.

Пусть линейный слой имеет вид

$$ W_{\mathrm{eff}} = W_q + UV^\top + S. $$

где:

- $W_q$ — основная квантованная матрица;
- $UV^\top$ — малоранговая поправка;
- $S$ — разреженная поправка.

Для входа

$$ X\in\mathbb R^{N\times d_{\mathrm{in}}}. $$

выход равен

$$ Y = XW_q^\top + XVU^\top + XS^\top. $$

Вычислительный граф можно представить так:

```text
                         ┌─ quantized GEMM с Wq ──────────┐
X ───────────────────────┤                                │
                         ├─ GEMM с V → GEMM с Uᵀ ─────────┼─ сумма → Y
                         │                                │
                         └─ sparse SpMM с Sᵀ ─────────────┘
```

### Базовая реализация без своего CUDA-кода

Концептуально так:

```python
def forward(x, w_quant, scales, u, v, s_sparse, bias=None):
    # Основная квантованная ветвь
    y = quantized_linear(x, w_quant, scales)

    # Low-rank ветвь:
    # x: [N, d_in]
    # v: [d_in, rank]
    # u: [d_out, rank]
    z = x @ v
    y_lr = z @ u.T

    # Sparse ветвь:
    # s_sparse: [d_out, d_in]
    y_sparse = torch.sparse.mm(s_sparse, x.T).T

    y = y + y_lr + y_sparse

    if bias is not None:
        y = y + bias

    return y
```
    
Здесь уже используются CUDA kernels, но они написаны на торче:

- один или несколько для quantized GEMM;
- два для для low-rank GEMM;
- ядро cuSPARSE для sparse matrix multiplication;
- ядра для сложения результатов.

cuSPARSE непосредственно поддерживает операции SpMM и форматы вроде CSR и COO. При этом библиотека ориентирована преимущественно на действительно разреженные матрицы - рабочий диапазон разреженности примерно от 70% до 99.9%.То есть сначала нужно сделать именно такую реализацию. 

# 2. Зачем вообще объединять ветви в собственный kernel

1. Например, для low-rank ветви создаётся промежуточная матрица Z=XV а затем полная матрица $Y_{\mathrm{lr}}=ZU^\top$
2. Для sparse-ветви создаётся ещё одна матрица $Y_{\mathrm{sparse}}=XS^\top$.
3. После этого GPU снова читает все три матрицы, чтобы сложить их - $Y=Y_q+Y_{\mathrm{lr}}+Y_{\mathrm{sparse}}$

Получаются дополнительные:

- запуски CUDA kernels;
- записи в глобальную память;
- чтения из глобальной памяти;
- промежуточные тензоры;
- синхронизации и управление памятью.

Особенно чувствительно это для сценариев где размер батча или число одновременно обрабатываемых токенов мало. Основной GEMM выполняется быстро, а накладные расходы на несколько маленьких операций становятся заметными. Идея fusion этих операций состоит в том, чтобы вычислить поправки и сразу добавить их к выходу, не создавая отдельно $Y_{\mathrm{lr}}$ и $Y_{\mathrm{sparse}}$.

# 3. Что такое CUDA kernel простыми словами

CUDA kernel — это функция, которая запускается одновременно в большом количестве GPU-потоков.Например, если выход имеет размер

$$ Y\in\mathbb R^{N\times d_{\mathrm{out}}}. $$

можно назначить каждому CUDA-потоку один элемент:

$Y_{no}$​.

Этот поток должен вычислить:

$$ Y_{no} \mathrel{+}= \sum_{k=1}^{r} Z_{nk}U_{ok} + \sum_{j:S_{oj}\neq0} X_{nj}S_{oj}. $$

CUDA организует потоки иерархически:

```text
grid
 ├─ block
 │   ├─ warp из 32 потоков
 │   ├─ warp из 32 потоков
 │   └─ ...
 ├─ block
 └─ ...
```

Потоки одного блока могут использовать общую быструю shared memory. Потоки исполняются группами по 32, называемыми warp. Если разные потоки одного warp выполняют разные ветви или циклы разной длины, часть потоков простаивает. Это называется warp divergence.


# 4. Самый реалистичный первый custom kernel



Первый разумный custom kernel должен решать гораздо более ограниченную задачу:

1. Основной quantized GEMM оставить существующей библиотеке.
2. Первый low-rank GEMM тоже оставить библиотеке - Z=XV.
3. Написать kernel, который за один проход выполняет - $Y \leftarrow Y + ZU^\top + XS^\top$
4. То есть он объединяет второй low-rank GEMM + sparse multiplication + сложение с основным выходом.

Схема:

```text
X ───────────── quantized GEMM ───────────────► Y
│                                                ▲
└─ GEMM с V ─► Z ──┐                            │
                    ├─ fused residual kernel ────┘
S в CSR ────────────┘
```


Это уже может убрать:

- тензор $Y_{\mathrm{lr}}$;
- тензор $Y_{\mathrm{sparse}}$;
- отдельные kernels сложения;
- повторные чтения этих тензоров.


Потом есть смысл попробовать INT4 GEMM. Но это сложная задача: - Tensor Cores, packed weights, scales, tiled loads, shared memory, register blocking.

## 4.1. Как хранится sparse-матрица

Для начала удобно использовать CSR:

row_ptr: [d_out + 1]
col_idx: [nnz]
values:  [nnz]


```text
S =
[ 0  2  0  0
  1  0  0  3
  0  0  4  0 ]
```

можно представить так:

```
row_ptr = [0, 1, 3, 4]
col_idx = [1, 0, 3, 2]
values  = [2, 1, 3, 4]
```


## 4.2. Упрощённый kernel

Это иллюстративный код, не производственная реализация:

```cpp
__global__ void add_lowrank_sparse_kernel(
    const half* __restrict__ x,          // [N, d_in]
    const half* __restrict__ z,          // [N, rank]
    const half* __restrict__ u,          // [d_out, rank]
    const int32_t* __restrict__ row_ptr, // [d_out + 1]
    const int32_t* __restrict__ col_idx, // [nnz]
    const half* __restrict__ s_values,   // [nnz]
    half* __restrict__ y,                // [N, d_out]
    int N,
    int d_in,
    int d_out,
    int rank
) {
    const int out_idx =
        blockIdx.x * blockDim.x + threadIdx.x;

    const int sample_idx =
        blockIdx.y * blockDim.y + threadIdx.y;

    if (out_idx >= d_out || sample_idx >= N) {
        return;
    }

    // Основная quantized-ветвь уже записала сюда результат.
    float acc = __half2float(
        y[sample_idx * d_out + out_idx]
    );

    // Low-rank поправка: dot(z[sample], u[out])
    for (int k = 0; k < rank; ++k) {
        const float z_value = __half2float(
            z[sample_idx * rank + k]
        );

        const float u_value = __half2float(
            u[out_idx * rank + k]
        );

        acc += z_value * u_value;
    }

    // Sparse поправка: dot(x[sample], S[out])
    const int begin = row_ptr[out_idx];
    const int end = row_ptr[out_idx + 1];

    for (int p = begin; p < end; ++p) {
        const int input_idx = col_idx[p];

        const float x_value = __half2float(
            x[sample_idx * d_in + input_idx]
        );

        const float s_value = __half2float(
            s_values[p]
        );

        acc += x_value * s_value;
    }

    y[sample_idx * d_out + out_idx] =
        __float2half(acc);
}
```

Запуск:

```cpp
dim3 block(16, 16);

dim3 grid(
    (d_out + block.x - 1) / block.x,
    (N + block.y - 1) / block.y
);

add_lowrank_sparse_kernel<<<grid, block, 0, stream>>>(
    x,
    z,
    u,
    row_ptr,
    col_idx,
    s_values,
    y,
    N,
    d_in,
    d_out,
    rank
);
```

Логически каждый поток отвечает за одну клетку выходной матрицы:

```
thread → один Y[sample, output]
```

Он:

1. читает уже посчитанный quantized output;
2. добавляет low-rank матричное перемножение;
3. проходит по ненулевым элементам соответствующей строки $S$;
4. один раз записывает итог.

# 5. Какие есть подводные камни в реализации

## 5.1. Разная длина sparse-строк

Допустим, четыре потока одного warp обрабатывают разные строки:

строка 0:  2 ненулевых элемента
строка 1: 37 ненулевых элементов
строка 2:  1 ненулевой элемент
строка 3: 19 ненулевых элементов

Первые потоки быстро заканчивают цикл, но должны ждать поток, который обрабатывает 37 элементов.

```text
thread 0: ██
thread 1: █████████████████████████████████████
thread 2: █
thread 3: ███████████████████
          └──────────────────────────────────────
          warp заканчивает работу здесь
```

Это один из вариантов warp divergence.

## 5.2. Нерегулярное чтение входа

Столбцы sparse-матрицы могут быть:

```
[7, 145, 802, 1993, ...]
```

Тогда соседние потоки читают из разных частей $X$. GPU не может объединить эти чтения в одну эффективную транзакцию памяти.

## 5.3. Слишком мало работы на поток

При маленьком rank и очень разреженной $S$ один поток делает мало арифметики. Тогда запуск kernel и обращения к памяти могут стоить больше, чем сами умножения.

## 5.4. Несбалансированная матрица

Если почти все ненулевые элементы сконцентрированы в небольшой части строк, одни thread blocks закончат быстро, а другие будут работать долго.


# 6. Как улучшить

Следующий уровень реализации — назначить один warp не одному элементу, а одной паре:

```
(sample, output row)
```

Все 32 потока warp совместно считают один результат.

Например:

warp lane 0:  rank indices 0, 32, 64, ...
warp lane 1:  rank indices 1, 33, 65, ...
...
warp lane 31: rank indices 31, 63, 95, ...


Каждый поток считает частичную сумму, после чего выполняется warp reduction:

$$ \mathrm{acc} = \sum_{\ell=0}^{31} \mathrm{partial}_\ell. $$

Аналогично делятся sparse-элементы строки.

Псевдокод:

```cpp
float partial = 0.0f;

for (int k = lane_id; k < rank; k += 32) {
    partial += z[k] * u[k];
}

for (int p = row_begin + lane_id; p < row_end; p += 32) {
    partial += x[col_idx[p]] * s_values[p];
}

partial = warp_reduce_sum(partial);

if (lane_id == 0) {
    y[index] += partial;
}
```

Такой подход лучше подходит, если:

- ранг равен 16, 32, 64;
- sparse-строки содержат десятки ненулевых элементов;
- размер батча мал.

Для большого батча можно назначать одному блоку одну строку $S$ и несколько объектов $X$, чтобы повторно использовать `col_idx` и `s_values`.

# 7. Как подключить kernel к PyTorch

Официальный путь PyTorch состоит из C++/CUDA custom operator. В типичном проекте появляются три файла:

```text
q_lr_sparse/
├── op.cpp
├── op_cuda.cu
└── setup.py
```

`op.cpp` выполняет:

- регистрацию операции;
- проверки размеров;
- проверки device и dtype;
- вызов CUDA launcher.

`op_cuda.cu` содержит:

- CUDA kernel;
- настройку block/grid;
- запуск kernel;
- проверку CUDA-ошибок.

Python вызывает операцию:

```python
torch.ops.q_lr_sparse.add_residual(
    x,
    z,
    u,
    row_ptr,
    col_idx,
    sparse_values,
    y,
)
```

PyTorch предоставляет официальный шаблон для C++/CUDA custom operators и регистрацию через диспатчер. Для инфренса реализация проще, потому что backward писать не требуется.

Детали которые надо чекать:

- запускать kernel на текущем CUDA stream PyTorch;
- проверять, что tensors находятся на одном GPU;
- контролировать contiguous layout;
- явно задавать поддерживаемые dtype;
- аккумулировать FP16/BF16 значения в FP32;
- отдельно тестировать `rank=0`, `nnz=0`, `batch=1`;
- сравнивать с эталонной PyTorch-реализацией.

# 9. Где здесь используются CUTLASS, cuSPARSE и cuSPARSELt

## CUTLASS

CUTLASS — библиотека шаблонов для высокопроизводительных GEMM. Она позволяет изменить epilogue - стадию, на которой результат матричного умножения масштабируется, складывается с bias и записывается в память. Современный CUTLASS поддерживает пользовательские epilogues.

Более продвинутая схема могла бы выглядеть так:

1. Посчитать - Z=XV.
2. Запустить основной quantized GEMM.
3. В его custom epilogue для каждого $Y_{no}$ дополнительно вычислить -$\sum_k Z_{nk}U_{ok}$
4. Сразу добавить bias и activation.

Тогда второй low-rank GEMM исчезает как отдельная операция. Но это сложнее обычного CUDA. CUTLASS обычно имеет смысл использовать тогда, когда простая Triton-реализация уже подтвердила пользу.

## cuSPARSE

Для произвольной unstructured sparse-матрицы сначала следует проверить cuSPARSE SpMM. Он поддерживает CSR, COO и смешанные типы вычислений.

## cuSPARSELt

Если sparse-компонента имеет аппаратно поддерживаемую структуру, например 2:4, полезен cuSPARSELt. В актуальной документации NVIDIA 50-процентная структурированная разреженность описана, в частности, как 2:4 для FP16, BF16, INT8 и ряда других типов.

|Структура $S$|Первый выбор|
|---|---|
|Произвольная CSR, 95–99% нулей|cuSPARSE|
|Block sparse|cuSPARSE или специализированный block kernel|
|2:4|cuSPARSELt|
|Очень малое фиксированное число элементов на строку|собственный warp kernel|
|Поддержка меняется для каждого объекта|grouped/custom kernel|

# 10. Первый прототип


$$ Y = XW_q^\top + ZU^\top + XS^\top, \qquad Z=XV. $$

То есть на инференсе:

Kernel 1: quantized_linear(X, Wq) → Y
Kernel 2: dense GEMM X @ V        → Z
Kernel 3: fused_residual(X, Z, U, S, Y)

Это компромисс:

- не нужно самостоятельно реализовывать INT4 GEMM;
- не создаются $Y_{\mathrm{lr}}$ и $Y_{\mathrm{sparse}}$;
- sparse support можно сжать один раз перед инференсом;
- rank и sparse-ветвь остаются раздельно управляемыми;
- kernel имеет относительно понятную семантику.

Полностью объединять $XV$ с остальными вычислениями сначала не стоит. Вектор $Z_n=XV$ используется для всех $d_{\mathrm{out}}$ выходов. Если пересчитывать его внутри каждого output tile, работа будет многократно дублироваться.


# 11. Как бенчмарчить?

Сначала нужно измерить обычную реализацию.

Nsight Systems показывает:

- сколько ядра запускается;
- есть ли паузы между ними;
- где выполняются синхронизации;
- сколько времени занимает CPU запуск.

Nsight Compute показывает характеристики конкретного ядра:

- пропускную способность памяти;
- загрузку вычислительных блоков;
- occupancy;
- warp divergence;
- число инструкций;

Свое ядро оправдано, когда профиль показывает что-либо из следующего:

- множество очень коротких последовательных ядер;
- значительное время на сложение ветвей;
- большие временные тензоры;
- sparse-операция плохо использует GPU;
- low-rank ветвь ограничена не FLOPS, а памятью;
- малый батч и оверхед сопоставим с вычислением.

Если большую часть времени занимает основной quantized GEMM, фьюз ветвей почти не изменит общую задержку.
